# Operational Usefulness

Operational usefulness is the standard for deciding whether Bob should do something.

If a feature does not reduce real operational pain, it should wait.

## Solving Operational Pain

Bob should help with problems like:

- missing PO references
- unclear install ownership
- duplicate customer records
- conflicting pricing sheets
- unresolved invoice disputes
- stale process documents
- scattered customer context

## Reducing Chaos

Bob reduces chaos by:

- finding sources
- showing conflicts
- linking related records
- separating facts from assumptions
- preserving corrections
- suggesting practical next actions

## Reconnecting Fragmented Context

Operational context often sits across CRM, finance, mailbox, spreadsheets and process notes.

Bob should reconnect that context carefully and show why the links exist.

Example:

ABC Ltd install status, PO thread, finance hold and duplicate CRM record belong together only if the source trail supports the link.

## Assisting Humans

Bob assists people by making the work clearer.

Bob should:

- explain what is known
- show what is uncertain
- suggest who should review something
- keep history inspectable
- support correction

Bob should not replace operational judgement.

## Avoiding AI For The Sake Of AI

Do not add AI-shaped features just because they are familiar.

Avoid:

- chatbot gimmicks
- broad assistant promises
- generic productivity claims
- opaque recommendations
- features that cannot show sources

## Enforceable Check

Before adding a feature, ask:

- What operational pain does this reduce?
- Which role benefits?
- Which source proves the answer?
- What uncertainty remains?
- Can a user correct it?
- Does it preserve trust?

If the answers are weak, the feature is not ready.
