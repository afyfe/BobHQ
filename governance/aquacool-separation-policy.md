# Aquacool Separation Policy

## Purpose

AskBob/BobHQ must remain operationally, technically and commercially separate from Aquacool and AquaOS work unless explicitly authorised.

## Core Principle

Aquacool knowledge may inform Bob only through:

- general professional experience
- generic operational patterns
- publicly shareable concepts
- explicitly authorised integrations

No Aquacool proprietary assets may be copied into BobHQ without explicit approval.

## Prohibited Without Explicit Authorisation

- Aquacool source code
- Aquacool databases
- Aquacool customer data
- Aquacool secrets or credentials
- Aquacool internal documents
- Aquacool deployment infrastructure
- Aquacool proprietary workflows
- Aquacool operational datasets

## Required Separation

BobHQ should maintain:

- separate repositories
- separate domains
- separate credentials
- separate infrastructure
- separate storage
- separate deployment pipelines
- separate Azure resources
- separate operational documentation

## Demo and Test Data

BobHQ demos should use:

- synthetic data
- fake SME datasets
- anonymised operational examples
- isolated test environments

Real Aquacool operational data must not be used without explicit approval.

## Integration Policy

Any future Aquacool integration must be:

- explicitly approved
- documented
- auditable
- scoped
- reversible

Aquacool should be treated as a potential future customer, not as BobHQ infrastructure.

## Pull Request Checklist

All BobHQ work should verify:

- No Aquacool code copied
- No Aquacool secrets included
- No Aquacool customer data used
- No Aquacool infrastructure dependency introduced
- Any Aquacool references are generic, anonymised or authorised

## MORSE and NAIB Reminder

Operational discipline and trust are core AskBob principles.

Separation reduces risk, preserves clarity and protects both organisations.
